# Design Review (advisory)

Generated by: awesome-copilot `azure-principal-architect` + `azure-verified-modules-bicep` personas, in fleet sub-agent.

## TL;DR
- Current hand-written modules are acceptable for the stated temporary closed-network GHES migration test; no design issue requires pausing deployment if the Defender P2 extension-suppression workstream is confirmed.
- The strongest near-term gaps are operational visibility and recoverability: no resource diagnostic settings / Log Analytics, and no GHES data-disk snapshot/backup workflow.
- AVM adoption is most attractive for Bastion and generic network primitives; keep the GHES VM module manual because GHES marketplace constraints make generic VM refactors risky.

## Section 2: WAF review

### Security
- ✅ GHES VM has no public IP; public exposure is limited to Bastion PIP and NAT Gateway PIP. NAT PIP is outbound-only.
- ✅ Both VMs disable password authentication and use SSH public keys.
- ✅ Bastion Standard has tunneling enabled and shareable links, IP connect, and file copy disabled; unauthorized use depends on Azure RBAC plus possession of VM SSH keys.
- ✅ Private DNS VNet link has `registrationEnabled: false`; manual A records are appropriate for this small, controlled test.
- ✅ GHES internet egress is effectively forced through the proxy today: GHES subnet has `defaultOutboundAccess: false`, no NAT Gateway, and an explicit `Deny-Out-Internet` after allowing tinyproxy and Azure DNS.
- ⚠️ `Allow-Out-Vnet` means GHES can initiate to any future VNet resource. This is not an internet escape today, but if strict egress evidence is required, narrow it or remove it after confirming Bastion return traffic remains stateful.
- ⚠️ Bastion access is not source-IP constrained; verify RBAC is least-privilege because the collected user egress IP is intentionally not enforced in NSGs.
- ⚠️ tinyproxy allows `10.0.0.0/8`; destination NSG currently limits inbound to `GhesSubnet:8888`, so risk is contained unless NSG/peering changes.
- ❌ No diagnostic settings on Bastion, NAT GW, NSGs, Public IPs, VNet, Private DNS, or VMs; boot diagnostics are enabled on VMs but not centralized.
- ✅ Platform-managed keys for OS/data disks are acceptable for this short-lived test; CMK would add unnecessary operational burden.

### Reliability
- ✅ Single VM / single-AZ-style design is acceptable for days-to-weeks validation, given the stated non-HA test constraint.
- ✅ Bastion Public IP is explicitly zone-redundant with zones `[1,2,3]`.
- ⚠️ Bastion host itself has no explicit `zones` property in the hand-written module; acceptable for test, but set/validate zones for production-grade Bastion.
- ⚠️ NAT Gateway and NAT PIP are zonal (`'1'`), which matches the cost-sensitive test posture but is not production-resilient.
- ❌ No GHES data disk backup/snapshot workflow. Since Azure Backup/ASR are unsupported for GHES, use a documented `az snapshot` process around key test milestones, ideally after quiescing/stopping GHES.
- ⚠️ DNS A records are manual while NIC private IPs are dynamic; fine for first deployment, but static private IPs reduce drift after NIC recreation.

### Performance Efficiency
- ✅ `Standard_E4s_v5` and a 200 GB Premium_LRS data disk are reasonable for migration validation workloads.
- ✅ GHES data disk caching is `ReadWrite`, matching the stated GHES guidance to check.
- ✅ `Standard_B2s` proxy VM and tinyproxy `MaxClients 100` are sufficient for controlled test egress.
- ⚠️ Accelerated Networking is disabled as required by GHES; note potential throughput/latency impact during large migration trials.

### Cost Optimization
- ⚠️ Rough always-on monthly burn is about **$600-$700/month** in an East US-like price region: Bastion Standard roughly **$210-$315** depending on scale-unit metering/region, NAT GW **~$33 + data**, GHES VM E4s_v5 Linux compute **~$184**, proxy B2s **~$30**, two Standard Public IPs **~$7**, Premium disks roughly **~$115** (GHES OS default ~400 GiB/P20, GHES 200 GiB data/P15, proxy OS/P4). Region and licensing can materially change this.
- ⚠️ Bastion is the dominant non-VM cost. A self-hosted SSH bastion VM would be cheaper but adds public-SSH hardening and operational burden; keep Azure Bastion unless cost pressure dominates.
- ⚠️ Stop/deallocate GHES and proxy VMs when idle to save compute; delete/redeploy Bastion and NAT Gateway between test windows if idle periods are long enough to justify the operational friction.
- ✅ No HA, no backups-as-a-service, no CMK, and no deployment locks are cost-appropriate for an ephemeral test.

### Operational Excellence
- ❌ No Log Analytics workspace or Azure Monitor diagnostic settings; troubleshooting depends on portal resource state, boot diagnostics, and in-guest logs.
- ✅ VM boot diagnostics are enabled for both VMs.
- ⚠️ Cloud-init is mostly idempotent for reruns (`mkdir -p`, service restart, full config rewrite), but `runcmd` normally runs only on first boot. Failures are visible only in `/var/log/cloud-init.log`, `/var/log/cloud-init-output.log`, and tinyproxy logs unless centralized logging is added.
- ⚠️ Tagging is basic/pass-through only; acceptable for a test, but add owner/expiry/cost-center tags if the environment may linger.
- ✅ No deployment stack/deny lock is fine for ephemeral infrastructure where fast teardown is desired.

## Section 3: Open questions for follow-up
- Does GHES first boot reliably initialize LUN 0 as `/data/user` for this marketplace image/version, or is a manual initialization step required?
- Will the Defender for Servers P2 RG-level exclusion/Free override prevent **all** unsupported extension auto-pushes for GHES, or only selected DSC/MDE paths?
- Is native-client Bastion tunneling strictly required? If portal-only access is enough, evaluate Developer/Basic Bastion or a temporary jump VM for cost reduction.
- Which region is final? Confirm Bastion/PIP/NAT zone support and update the burn estimate with regional prices.

## Section 4: Recommended action items (prioritized)

1. **MUST fix before deploy**: Confirm the Defender P2 RG-scope override/exclusion prevents unsupported extensions from being pushed to the GHES VM. Blocking only if unconfirmed.
2. **SHOULD address before prod**: Add centralized diagnostic settings + Log Analytics for Bastion, NSGs/flow logs, NAT/Public IPs, VNet, and VM guest/boot signals where supported.
3. **SHOULD address before prod**: Document and test a GHES disk snapshot procedure (`az snapshot`) before/after migration milestones; quiesce/stop GHES when consistency matters.
4. **SHOULD address before prod**: Rework availability design for production: zone-aware Bastion/VMs, HA GHES topology if supported, non-zonal NAT avoidance, and tested restore path.
5. **SHOULD address before prod**: Tighten egress if audit evidence is required: reduce `Allow-Out-Vnet`, narrow tinyproxy `Allow`, and consider static private IPs for DNS records.
6. **NICE to have**: Replace `bastion.bicep` with pinned AVM `network/bastion-host` once parameters are mapped.
7. **NICE to have**: Use AVM for network primitives in a hybrid refactor while preserving exact NSG/subnet semantics.
8. **NICE to have**: Add owner/expiry tags and an explicit stop/delete runbook to avoid paying Bastion/NAT/VM compute after the test window.
